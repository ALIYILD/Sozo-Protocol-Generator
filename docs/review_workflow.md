# SOZO Generator — Clinical Review and Governance Workflow

## Purpose

This document defines the governance and human review processes that apply to all
documents generated by the `sozo_generator` platform. Generated documents are clinical
materials; they must meet SOZO Brain Center's quality and safety standards before use
with patients.

---

## When Human Review Is Required

Human clinical review is mandatory before any generated document is used with a patient
in any of the following situations:

### 1. Review Flags Are Present

Any document that contains one or more of the following `ReviewFlag` values must be
reviewed by a qualified clinician before clinical release:

| Flag | Meaning |
|---|---|
| `missing_primary_source` | A claim has no directly supporting published reference |
| `contradicting_sources` | Two or more references make conflicting claims |
| `indirect_evidence_only` | Claims are supported only by indirect or analogical evidence |
| `pilot_data_only` | The highest available evidence is pilot or feasibility study data |
| `off_label_not_flagged` | An off-label stimulation target is missing required consent markup |
| `template_carryover` | Content appears carried over from another condition without adaptation |
| `incomplete_safety` | The safety notes section is absent or incomplete |
| `no_validated_scale` | An assessment tool lacks a validated psychometric reference |
| `placeholder_text` | Unreplaced template placeholder text is present |
| `missing_section` | A required document section is absent |

Flags are visible in QA reports generated by:

```bash
PYTHONPATH=src python -m sozo_generator.cli.main qa report \
    --condition {slug} --format markdown
```

### 2. QA Completeness Below 80%

The QA system marks a condition as `NOT READY` when `overall_completeness < 0.80`
(fewer than 12 of 15 expected documents pass). No documents for such a condition should
be used clinically until the completeness threshold is met.

### 3. New Condition Onboarding

The first version of every newly added condition requires review by the Clinical Lead
before the condition's documents enter active clinical use. This applies regardless
of QA pass status.

### 4. Protocol Changes After New Evidence

Whenever a published study material changes the protocol parameters, stimulation targets,
or responder criteria for a condition, the affected documents must be reviewed and
re-approved before the updated versions are used with patients.

### 5. Evidence Quality of LOW or VERY_LOW

If `ConditionSchema.overall_evidence_quality` is `EvidenceLevel.LOW` or `EvidenceLevel.VERY_LOW`,
the entire condition's document set requires senior clinician sign-off before clinical use.
The generated documents will carry a visible notice in the header.

## Off-Label Use Consent Requirements

All neuromodulation modalities used at SOZO Brain Center (tDCS, TPS, taVNS, CES) are
applied off-label for most neurological and psychiatric indications.

### Before the First Session

1. The treating clinician must present the patient with a written off-label informed consent
   document that includes:
   - Name of the modality and stimulation target
   - The off-label nature of the treatment
   - Current evidence level and known limitations
   - Known risks, adverse effects, and stopping criteria
   - Alternative treatment options

2. The patient must sign and date the consent form.

3. The signed consent form must be filed in the patient's clinical record before
   the first treatment session begins.

4. The consent must be re-obtained if:
   - A new modality is added to the patient's protocol
   - The stimulation target changes
   - A new condition indication is added

### TPS-Specific Requirement

All Transcranial Pulse Stimulation (TPS) applications require **explicit off-label informed
consent AND Doctor authorization** prior to first session, per `SHARED_GOVERNANCE_RULES`
in `conditions/shared_condition_schema.py`. This requirement applies regardless of
condition or clinical tier.

## Two-Tier Clinical Review Model

`sozo_generator` generates documents for two distinct clinical tiers with different
governance requirements.

### Fellow Tier — Supervised Practice

**Who:** Clinical Fellows undergoing supervised training at SOZO Brain Center.

**Supervision requirements:**
- Fellows may not independently modify treatment plans without Doctor authorization.
- All protocol parameter changes require Doctor approval before implementation.
- Fellows must document every session in the patient's record before leaving the clinic.
- Adverse events must be reported to the supervising Doctor immediately and documented
  within 24 hours.
- Fellows use the 7 Fellow-tier documents; they do not administer the 6-Network Bedside
  Assessment independently.

**Documents generated for Fellows:**
- Clinical Examination Checklist
- Phenotype Classification
- Responder Tracking
- Psychological Intake / PRS Baseline
- SOZO Clinical Handbook
- FNON All-in-One Protocol
- Evidence-Based Protocol

### Partners Tier — Independent Practice

**Who:** Credentialed SOZO Partners with full FNON training and clinical authorization.

**Practice requirements:**
- Partners may independently administer all supported modalities and protocols.
- Partners conduct the full 6-Network Bedside Assessment and use all 8 Partners-tier documents.
- Partners retain responsibility for off-label consent documentation.
- Adverse events must be documented within 24 hours and escalated per clinic protocol.
- Partners may propose protocol parameter changes based on patient response; these must be
  documented in the patient record and are subject to periodic peer review.

**Additional document generated for Partners:**
- 6-Network Bedside Assessment (assesses all 6 FNON networks: DMN, CEN, SN, SMN, LIMBIC, ATTENTION)

## Evidence Gap Documentation Process

When evidence is missing or insufficient for a clinical claim, the gap must be explicitly
documented rather than silently omitted or masked.

### Recording an Evidence Gap

1. Add a descriptive string to `ConditionSchema.evidence_gaps` in the condition generator:

   ```python
   evidence_gaps=[
       "No RCT data available for tDCS in {condition} as of 2024; "
       "current protocol based on consensus and pilot data (PMID: 12345678).",
   ]
   ```

2. Set the corresponding `review_flags` entry:

   ```python
   review_flags=["pilot_data_only"]
   ```

3. Set the `overall_evidence_quality` to reflect the true state:

   ```python
   overall_evidence_quality=EvidenceLevel.LOW,
   ```

### How Gaps Appear in Documents

The content assembler inserts the following notice in sections where evidence is insufficient:

> **REQUIRES CLINICAL REVIEW — Evidence insufficient:** [claim text]

This notice is visible in the generated DOCX file. The treating clinician must
acknowledge this notice before using the information clinically.

### Gap Review Cycle

Evidence gaps are reviewed on a rolling basis:
- New PubMed evidence is ingested periodically using `ingest-evidence ingest`.
- If new evidence resolves a documented gap, the gap entry is removed, the reference is
  added, and the `review_flags` entry is cleared.
- The updated condition documents are rebuilt and re-reviewed before clinical release.

## How to Update a Protocol After New Evidence

When a new publication materially changes a stimulation protocol (parameters, target,
session count, responder criteria), follow this process:

### 1. Verify the Evidence

Confirm the PMID is real and the study is applicable:
- Visit `https://pubmed.ncbi.nlm.nih.gov/{PMID}/`
- Confirm the study population is relevant to the SOZO patient profile
- Confirm the evidence type (RCT, cohort, pilot, etc.) and assign the correct `EvidenceLevel`

### 2. Update the Condition Generator

In `src/sozo_generator/conditions/generators/{slug}.py`:

- Add the new reference to the `references` list with all required fields.
- Update the affected `ProtocolEntry.parameters` (intensity, duration, session count, etc.).
- Update the `ProtocolEntry.evidence_level` if the new study elevates it.
- Update `ProtocolEntry.rationale` to reference the new evidence.
- If the target changes, update both `ProtocolEntry.target_region` and the corresponding
  `StimulationTarget`.
- Update `ConditionSchema.overall_evidence_quality` if warranted.
- Remove any `evidence_gaps` entries that the new study resolves.
- Remove any `review_flags` that are no longer applicable.

### 3. Rebuild Documents

```bash
PYTHONPATH=src python -m sozo_generator.cli.main build condition \
    --condition {slug} --tier both --doc-type all
```

### 4. Run QA

```bash
PYTHONPATH=src python -m sozo_generator.cli.main qa report \
    --condition {slug} --format markdown
```

Confirm the condition shows `PASS` and no new flags have appeared.

### 5. Clinical Review

The updated documents must be reviewed by the Clinical Lead before the new protocol
version enters clinical use. The reviewer must confirm:

- The new evidence is applicable to the SOZO patient population.
- The protocol parameters are within safe operating ranges.
- Off-label consent documentation reflects the updated parameters.
- Existing patients on the old protocol are notified if their treatment plan changes.

### 6. Version Control

Document the protocol update in the condition generator's module docstring
with the date, changed fields, and PMID of the supporting evidence:

```python
"""
{Condition} condition schema builder.

Change log:
  2025-06-01: Updated DLPFC tDCS intensity from 1.5 mA to 2.0 mA
              based on PMID 12345678 (RCT, n=60, 2025).
"""
```

## Governance Rules (All Tiers)

The following rules apply to all clinical staff at SOZO Brain Center, regardless of tier:

1. No Fellow or Clinical Assistant may independently modify treatment plans without Doctor authorization.
2. All TPS applications require explicit off-label informed consent and Doctor authorization.
3. Adverse events must be documented within 24 hours and escalated per clinic protocol.
4. Treatment records must be updated at every session.
5. Informed consent must be obtained and documented before initiating any modality.

These rules are defined in `conditions/shared_condition_schema.py` as
`SHARED_GOVERNANCE_RULES` and are included in the handbook and protocol documents
for all 15 conditions.
